## 一、核心业务流程：奖学金评定（最关键链路）

这是系统的核心价值链，从批次创建到结果确认的完整流程：

```
创建批次 → 开始申请 → 开始评审 → 开始公示 → 完成评定
```

### 1.1 批次生命周期管理

| HTTP方法 | 路径 | 说明 |
|----------|------|------|
| `POST` | `/evaluation-batch` | 管理员创建评定批次 |
| `PUT` | `/evaluation-batch/start/{id}` | **开始申请** — 批次从未开始 → 申请中 |
| `PUT` | `/evaluation-batch/start-review/{id}` | **开始评审** — 批次从申请中 → 评审中 |
| `PUT` | `/evaluation-batch/start-publicity/{id}` | **开始公示** — 批次从评审中 → 公示中 |
| `PUT` | `/evaluation-batch/close/{id}` | **完成评定** — 批次从公示中 → 已完成 |
| `GET` | `/evaluation-batch/available` | 学生/导师查看当前可申请批次 |

### 1.2 评定计算链

| HTTP方法 | 路径 | 说明 |
|----------|------|------|
| `POST` | `/evaluation-result/evaluate/{batchId}` | ⭐ **一键评定**（统一事务：计算评分 → 生成排名 → 分配奖项） |
| `POST` | `/evaluation-result/calculate/{batchId}` | 分步：计算批次下所有申请的评分 |
| `POST` | `/evaluation-result/generate-ranks/{batchId}` | 分步：生成院系和专业排名 |
| `POST` | `/evaluation-result/generate/{batchId}` | 分步：根据排名自动分配奖项等级和金额 |
| `PUT` | `/evaluation-result/confirm/{id}` | 公示无异议后确认结果 |
| `PUT` | `/evaluation-result/adjust/{id}` | 管理员手动调整奖项等级 |
| `PUT` | `/evaluation-result/object/{id}` | 将评定结果标记为有异议状态 |
| `GET` | `/evaluation-result/export` | 导出评定结果为 Excel |

## 五、系统关键设计要点

### 5.1 权限分层

| 角色 | 权限范围 |
|------|----------|
| `ROLE_ADMIN` | 全部数据、全部操作 |
| `ROLE_TUTOR` | 仅限本人指导学生的数据，通过 `LoginUser` 动态过滤 |
| `ROLE_STUDENT` | 仅限本人的数据 |

### 5.2 统一事务

`POST /evaluation-result/evaluate/{batchId}` 通过 `EvaluationWorkflowService` 将计算评分 → 生成排名 → 分配奖项串在统一事务中，中途失败可整体回滚。

### 5.3 审核双通道

科研成果（论文/专利/项目/竞赛）必须先通过导师或管理员审核（`auditStatus=APPROVED`），才能被奖学金申请关联。

### 5.4 批次状态机

```
未开始 → 申请中 → 评审中 → 公示中 → 已完成
  │        │        │        │
  └────────┴────────┴────────┘
         状态流转由管理员手动触发
```

### 5.5 申诉闭环

- 学生只能在公示中状态（`PUBLICITY`）下提交异议
- 管理员处理后形成闭环
- 同一学生同一申请不可重复提交处理中的申诉

## 七、高并发异步优化分析

### 7.1 异步能力现状

| 能力 | 状态 | 说明 |
|------|:--:|------|
| `@EnableAsync` | ✅ 已启用 | `ScholarshipApplication.java:40` |
| `@Async` 使用 | ⚠️ 仅1处 | `TokenBlacklistServiceImpl.java:94`（Redis token清理） |
| 自定义线程池 | ❌ 缺失 | 使用默认 `SimpleAsyncTaskExecutor`（无线程管理，高风险） |
| 消息队列 | ❌ 未使用 | 开发配置中已注释：`RocketMQ 在本项目中未实际使用` |
| 定时任务 | ❌ 无 | 全项目 0 处 `@Scheduled` |
| Redis 分布式锁 | ✅ 有 | `RedisLockSupport.java`（评定锁30分钟、申请锁10秒） |
| 操作日志写入 | ❌ 缺失 | 有实体/枚举/查询接口，无写入 AOP/拦截器 |
| 邮件/短信/WebSocket | ❌ 无 | 仅站内信（`sys_notification` 表），前端轮询 |
| 数据库连接池 | Druid | 生产: max-active=100, 慢SQL阈值5s |
| Redis 连接池 | Lettuce | 生产: max-active=50 |

### 7.2 性能瓶颈分布

| 等级 | 接口 | 瓶颈描述 | 位置 |
|:--:|------|------|------|
| **严重** | `POST /evaluate/{batchId}` 一键评定 | 3步串行在同一个 `@Transactional` 中，Redis锁30分钟，连接持有超长 | `EvaluationWorkflowServiceImpl.java:37` |
| **严重** | `POST /batch/import/students` 学生导入 | 逐行 `selectOne` + `insert`，1000条=2000次DB调用 | `BatchImportServiceImpl.java:35` |
| **严重** | `POST /course-score/import` 成绩导入 | 逐行解析Excel + 逐条insert/update，长事务混合读写 | `CourseScoreServiceImpl.java:304` |
| **严重** | `POST /application/submit` 提交申请 | Redis锁 + INSERT申请 + 替换成果关联（先删后插），全在一个事务 | `ScholarshipApplicationServiceImpl.java:103` |
| **高** | `GET /course-score/export` 导出 | 全量查询无分页，大数据量OOM风险 | `CourseScoreController.java:208` |
| **高** | `GET /evaluation-result/export` 导出 | 全量查询无分页 | `EvaluationResultController.java:287` |
| **高** | 全局操作日志写入 | 功能完全缺失 | 无AOP/拦截器实现 |
| **高** | Druid wall filter 冲突 | 生产环境 `delete-permit: false` 与 `remove()` 物理删除冲突 | `application-prod.yml:44` |
| **中** | 通知发送 | 仅站内信DB写入，无邮件/短信，前端轮询 | `SysNotificationServiceImpl.java` |

### 7.3 技术方案对比：@Async vs RocketMQ

| 维度 | @Async + 线程池 | RocketMQ |
|------|:---------------:|:--------:|
| 消息持久化 | ❌ 内存级，进程崩溃任务丢失 | ✅ 消息持久化到磁盘 |
| 重试机制 | ❌ 需手动实现 | ✅ 自带消费重试（默认16次） |
| 削峰填谷 | ⚠️ 受线程池队列上限限制 | ✅ 消息堆积天然削峰 |
| 顺序保证 | ❌ 无法保证 | ✅ 顺序消息（MessageQueue级别） |
| 延时任务 | ❌ 需手动实现 | ✅ 内置18个延时级别 |
| 运维复杂度 | ✅ 零额外组件 | ⚠️ 需部署 NameServer + Broker |
| 适合场景 | 轻量、允许丢失的操作 | **关键业务、不允许丢失的操作** |

**结论：按场景分路线**
```
关键业务（不能丢） → RocketMQ（评定、导入、日志）
非关键业务（可丢弃）→ @Async（通知发送）
纯 DB 优化（不需异步）→ 批量操作改造（导入内部SQL优化）
```

### 7.4 适合 RocketMQ 的接口

#### ⭐⭐⭐ 一键评定 `POST /evaluation-result/evaluate/{batchId}`

**用MQ理由**: 系统最核心操作，执行时间数分钟，服务重启不能丢失评定请求；MQ 持久化+自带重试+支持幂等消费

**MQ 消息设计**:
```
Topic: EVALUATION_BATCH
Tag: evaluate
Body: { "batchId": 123, "taskId": "uuid", "operatorId": 1 }
消费重试: 最多3次
```

**链路**: Controller → `rocketMQTemplate.asyncSend()` → 立即返回 taskId → Consumer 执行[计算→排名→分配] → 更新 Redis 进度 → 发通知

#### ⭐⭐⭐ 操作日志写入（全链路 AOP）

**用MQ理由**: 经典 MQ 场景——AOP 切面织入 → asyncSend MQ → Consumer 批量落库；零侵入、削峰、不阻塞主业务；后续可接入 ELK、数据分析

```
Topic: OPERATION_LOG
Tag: login / user / evaluation / system
Body: { "operatorId": 1, "operationType": 3, "module": "评定管理",
        "description": "一键评定批次123", "executionTime": 1234, "ip": "..." }
```

**实现**: `@OperationLog` 注解 → `OperationLogAspect` 切面 → `RocketMQTemplate.asyncSend()` → `OperationLogConsumer` 落库

#### ⭐⭐ 成绩批量导入 `POST /course-score/import`

**用MQ理由**: 大Excel数千行，写入耗时数十秒；消息持久化保证不丢失；Excel解析同步完成（内存操作），DB写入异步消费

```
Topic: COURSE_SCORE_IMPORT
Tag: import
Body: { "taskId": "uuid", "studentId": 1, "rows": [...解析后的成绩列表...] }
```

#### ⭐⭐ 学生批量导入 `POST /batch/import/students`

**用MQ理由**: 同成绩导入，重IO操作，MQ削峰+持久化

```
Topic: STUDENT_IMPORT
Tag: import
Body: { "taskId": "uuid", "students": [...解析后的学生列表...] }
```

#### ⭐ 通知发送（@Async 即可，MQ 为未来扩展预留）

当前仅站内信DB写入，用 `@Async` 即可。后续接入邮件/短信时，MQ 是最佳实践。

```
Topic: NOTIFICATION
Tag: submit_application / evaluation_complete / appeal_handled
```

### 7.5 不推荐接入 MQ 的接口

| 接口 | 原因 | 替代方案 |
|------|------|----------|
| `GET */export` 导出 | 用户同步等待下载 | 分页查询 + EasyExcel 流式写入 |
| `POST /evaluate/calculate/{id}` 评分计算(单步) | 已有分布式锁+批量写入，事务性强 | 保持同步，优化批量SQL |
| `POST /evaluate/generate-ranks/{id}` 排名生成(单步) | 纯内存计算，耗时不高 | 保持同步 |
| `GET /course-score/weighted-average/{id}` 加权平均分 | 轻量查询聚合 | 保持同步 |
| `GET /moral-performance/total-score/{id}` 德育总分 | 轻量查询聚合 | 保持同步 |

#### 自定义线程池配置

```java
@Configuration
public class AsyncThreadPoolConfig {

    // 核心异步任务线程池：评定、导入、导出
    @Bean("taskExecutor")
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(4);
        executor.setMaxPoolSize(8);
        executor.setQueueCapacity(50);
        executor.setKeepAliveSeconds(60);
        executor.setThreadNamePrefix("async-task-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()); // 关键任务不丢弃
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(30);
        executor.initialize();
        return executor;
    }

    // 日志/通知专用线程池：轻量、快速、允许丢弃
    @Bean("logNotifyExecutor")
    public ThreadPoolTaskExecutor logNotifyExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);
        executor.setMaxPoolSize(4);
        executor.setQueueCapacity(200);        // 大队列缓冲峰值
        executor.setKeepAliveSeconds(30);
        executor.setThreadNamePrefix("log-notify-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.DiscardOldestPolicy()); // 允许丢弃
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(10);
        executor.initialize();
        return executor;
    }
}
```

#### Redis 异步任务进度管理

```
Key:  async:task:{taskId}
Fields:
  taskId     -> 任务唯一标识
  status     -> PENDING / RUNNING / COMPLETED / FAILED
  progress   -> 0-100 (百分比)
  totalCount -> 总处理数
  doneCount  -> 已完成数
  message    -> 当前步骤描述
  errorMsg   -> 失败原因
  result     -> JSON 序列化结果数据
TTL: 3600 秒（1小时后自动清理）
```

**前端轮询接口**: `GET /async-task/{taskId}/progress`
