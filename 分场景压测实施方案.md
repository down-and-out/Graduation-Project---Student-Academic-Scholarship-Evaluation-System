# 奖学金评审系统 — 分场景压测实施方案

## Context

当前系统已完成缓存、限流、异步线程池、索引优化等基础建设，但缺少真实压力下的性能数据支撑后续优化决策。需要在引入 RocketMQ 之前，先对 4 个核心热点接口做分场景压测，用数据区分瓶颈归属（数据库/Redis/线程池/业务计算），指导后续优化方向。

## 目标接口

| 接口 | 特点 |
|------|------|
| `POST /application/submit` | 同步、Redis锁、事务、7表读写 |
| `POST /evaluation-result/evaluate/{batchId}` | 异步(@Async)、13表、复杂计算链路 |
| `GET /evaluation-result/page` | 同步、@Cacheable(180s)、2表读 |
| `GET /evaluation-result/export` | 同步全量、EasyExcel、OOM风险 |

## 实施步骤

### 步骤 1：添加 Actuator 监控依赖

**文件**: `scholarship-admin/pom.xml`

在 `<dependencies>` 中添加 `spring-boot-starter-actuator`。当前项目没有 Actuator，线程池活跃数/队列长度无法通过 JMX/HTTP 获取。

在 `application-dev.yml` 中添加 `management.endpoints` 配置，暴露 `health,metrics,prometheus,threaddump` 端点。

### 步骤 2：暴露线程池指标为 Actuator Metric

**文件**: `scholarship-admin/src/main/java/com/scholarship/config/AsyncTaskConfig.java`

为 3 个线程池 (`evaluationTaskExecutor`, `exportTaskExecutor`, `batchImportTaskExecutor`) 各注册一个 `MeterBinder` Bean，暴露以下 Gauge：
- `executor.{name}.active.count` — 当前活跃线程数
- `executor.{name}.queue.size` — 等待队列长度
- `executor.{name}.pool.size` — 当前线程池大小
- `executor.{name}.completed.tasks` — 已完成任务数

这些指标会出现在 `/actuator/metrics/executor.*` 端点，可通过 curl 轮询采集。

### 步骤 3：准备压测数据

**文件**: `stress-test/scripts/prepare-data.sql`

编写存储过程批量生成：
- 50~200 名测试学生（`sys_user` + `student_info` + `course_score` + `moral_performance`）
- 在同一申请批次下预填 `evaluation_result` 数据（供 page/export 压测用）
- 准备一个含申请数据的小批次（10条）和一个中批次（100条）供 evaluate 压测

### 步骤 4：编写 JMeter 测试计划

**文件结构**:
```
stress-test/jmeter/
├── login-flow.jmx              # 可复用登录片段（JMeter Test Fragment）
├── application-submit.jmx      # 接口1 申请提交
├── evaluation-execute.jmx      # 接口2 评审执行
├── result-page.jmx             # 接口3 分页查询
└── result-export.jmx           # 接口4 导出
```

**登录前置处理**：
系统使用 RSA 加密密码登录。在 JMeter 中实现方案：
1. 新增一个公开端点 `GET /auth/public-key`（在 AuthController 中加一个简单方法返回 RSA 公钥的 Base64 编码），或
2. 更简单：在 `stress-test/scripts/` 下提供 `generate-tokens.sh`，用 curl 预先生成一批 Token 写入 CSV 文件，JMeter 直接读取 Token 绕过登录

**方案2**是更推荐的，因为压测本身不应包含登录接口的瓶颈干扰。脚本逻辑：
- 调用 Spring Security 的 `PasswordEncoder` 或直接用 BCrypt 密文模拟
- 实际上可以用一个已有的测试账号 `admin/admin123` 获取 Token，然后用 JMeter `CSV Data Set Config` 分发

**每个 .jmx 的 Thread Group 结构**（统一模式）：

```
Test Plan
├── setUp Thread Group (登录获取Token，写入属性)
│   └── HTTP Request: POST /auth/login
│       └── JSON Extractor: token
├── Thread Group: 基准测试 (1线程, loop=20)
├── Thread Group: 负载-50并发 (50线程, ramp-up=10s, duration=300s)
├── Thread Group: 负载-100并发 (100线程, ramp-up=15s, duration=300s)
├── Thread Group: 压力测试 (200线程, ramp-up=20s, duration=120s)
└── tearDown Thread Group (输出统计)
```

**接口1 submit 特有场景**：
- 同批次重复提交（同一个 studentId 并发 → 验证幂等和锁竞争）
- 峰值突刺（Ultimate Thread Group: 0→50→200→0 的阶梯式并发）

**接口2 evaluate 特有场景**：
- 小/中/大批次（参数化 batchId = 3个不同规模的批次）
- 重复触发（同一 batchId 并发 → 验证 `reusedActiveTask` 机制）
- 多管理员并发不同批次（模拟真实场景）
- 线程池队列满载（连续触发10个批次 → 验证 CallerRunsPolicy 行为）

**接口3 page 特有场景**：
- 不同分页深度（page=1, 10, 50, 250 → 对比 offset 对性能的影响）
- 缓存命中 vs 未命中（相同参数重复请求 vs 随机参数）

**接口4 export 特有场景**：
- 50/500/5000 条数据的导出耗时对比
- 限流验证（连续请求 → 确认第4次起返回 429）

### 步骤 5：编写指标采集脚本

**文件**: `stress-test/scripts/monitor.sh`

一个 Bash 脚本，在压测期间以 1 秒间隔采集以下指标并追加到 CSV 文件：

```csv
timestamp, druid_active, druid_wait, redis_rtt_ms, executor_eval_active, executor_eval_queue, tomcat_busy, cpu_pct, mem_pct, gc_fgct
```

数据来源：
- **Druid**: `curl -s http://localhost:8080/druid/datasource.json` → 解析 `ActiveCount`, `WaitThreadCount`
- **Redis RTT**: `redis-cli --latency` 或 `redis-cli PING` 配合计时
- **线程池**: `curl -s http://localhost:8080/actuator/metrics/executor.evaluation.active.count` 等
- **Tomcat**: `curl -s http://localhost:8080/actuator/metrics/tomcat.threads.busy`
- **CPU/内存**: `top -b -n1` 或 Windows `typeperf`
- **GC**: `jstat -gcutil <pid> 1000`（需要 Java 进程 PID）

另外单独一个脚本 `stress-test/scripts/slow-sql-monitor.sh`，在压测后从 Druid `/druid/sql.json` 导出慢 SQL 统计。

### 步骤 6：编写自动化执行脚本

**文件**: `stress-test/scripts/run-benchmark.sh`

按顺序执行：
```
Phase 1: 基准测试 (各接口 1 线程, 20 次)
Phase 2: 50 并发负载 (5 分钟)
Phase 3: 100 并发负载 (5 分钟)
Phase 4: 200 并发压力 (2 分钟)
Phase 5: 稳定性测试 (30 线程, 30 分钟)
```

每个 Phase 前启动 `monitor.sh` 后台采集，Phase 结束后 `kill` 采集进程。结果 `.jtl` 文件输出到 `stress-test/results/<timestamp>/`。

### 步骤 7：瓶颈分类分析

压测完成后，结合以下数据做瓶颈分类：

| 指标组合 | 判定 |
|----------|------|
| Druid ActiveCount 持续接近 max-active(50) + 等待线程 > 0 | **数据库连接池瓶颈** |
| P95 高 + Druid 连接空闲充足 + Redis RTT 正常 | **业务同步计算瓶颈** |
| Redis RTT > 5ms + Lettuce 连接池满 | **Redis 瓶颈** |
| executor queue size 持续等于 capacity + CallerRuns 导致 RT 飙升 | **线程池瓶颈** |
| 429 比例高但取消限流后 TPS 线性增长 | **限流器正常防护（非瓶颈）** |
| 大分页 offset 增长时 RT 非线性增长 | **MySQL 深分页瓶颈** |
| GC FGC > 0 + 堆内存 > 85% | **JVM 堆内存瓶颈** |

分析结果输出为 `stress-test/results/analysis-report.md`。

## 验证方式

1. 执行 `prepare-data.sql` 后确认 `SELECT COUNT(*) FROM student_info` >= 50
2. 启动应用后确认 `/actuator/health` 返回 200
3. 确认 `/actuator/metrics/executor.evaluation.active.count` 有值
4. 单线程执行 `application-submit.jmx` 确认 200 响应
5. 单线程执行 `evaluation-execute.jmx` 确认 taskId 正确返回
6. 单线程执行 `result-page.jmx` 确认分页数据正确
7. 单线程执行 `result-export.jmx` 确认文件下载正常
8. 运行 `monitor.sh` 确认各项指标均有数据输出
9. 完整执行 5 个 Phase 后检查 `results/` 目录下的 .jtl 和 .csv 文件
