## 三、可优化的点

### 🐛 脚本/工具层面

| # | 问题 | 涉及文件 | 优化建议 |
|---|------|---------|---------|
| 1 | `generate-tokens.sh` JSON 解析 | `stress-test/scripts/generate-tokens.sh:66-70` | `extract_json_number` 改为兼容 `"key":"number"` 和 `"key":number` 两种格式 |
| 2 | `generate-tokens.sh` 路径问题 | `stress-test/scripts/generate-*.sh` | 输出目录应使用绝对路径或 `SCRIPT_DIR` 基准，而非相对路径 `../data/` |
| 3 | `prepare-data.sql` 有无效 SQL | `stress-test/scripts/prepare-data.sql:248` | `GROUP BY batch_code` 应改为子查询关联 `evaluation_batch` 或直接 `GROUP BY batch_id` |
| 4 | `prepare-data.sql` BCrypt hash 错误 | `stress-test/scripts/prepare-data.sql:52` | 学生账号的密码 hash 应通过 `BCryptPasswordEncoder` 真实生成，而非写死无效 hash |
| 5 | `evaluation-execute.jmx` 兼容性 | `stress-test/jmeter/evaluation-execute.jmx:105,120,135` | 升级 JMeter 至 5.7+ 或将 `floatProp` 改为 `stringProp` |
| 6 | `application-submit.jmx` Groovy 条件 | `stress-test/jmeter/application-submit.jmx:71,110` | `submit-conflict` 线程组在条件为 false 时仍空轮询消耗资源 → 使用 `disabled` 属性或用模块控制器 |

### ⚙️ 后端配置层面

| # | 问题 | 涉及文件 | 优化建议 |
|---|------|---------|---------|
| 7 | 登录限流配置分散 | `application-dev.yml` security.login.* | 压测环境应可通过单个环境变量一键关闭所有限流 |
| 8 | `/actuator/health` 需要认证 | Spring Security 配置 | 放开 `/actuator/health` 的匿名访问，使预检脚本能正常工作 |
| 9 | 导出接口并发能力差 | `evaluation-result/export` | 50% 请求连接被拒绝 → 考虑异步导出 / 增加 export 线程池 / 引入排队 |
| 10 | Redis IP 封锁持久化 | `LoginAttemptServiceImpl` | 压测环境重启后旧 IP 封锁仍在 Redis 中生效 → 增加压测 profile 自动清理 |

### 🏗️ 压测流程层面

| # | 问题 | 优化建议 |
|---|------|---------|
| 11 | Token 生成太耗时 | 200 个学生逐个登录需 2+ 分钟 → 实现批量 Token 颁发接口 `/auth/token/batch` |
| 12 | Phase 7(稳定性)未执行 | 30 分钟稳定性压测跳过了 → 单独安排执行 |
| 13 | 混合流量 JVM segfault | JMeter 默认堆内存不够 100 线程 → `-Xmx2g` 或分拆 ThreadGroup |
| 14 | 自动化报告依赖 Python3 | `analyze-results.sh` 未成功运行 → 改为纯 Shell awk 版本增强鲁棒性 |
| 15 | Windows 兼容性差 | 脚本多处假设 Linux（管道、tail 参数、进程管理、`pkill`）→ 增加 OS 检测分支或提供 bat 版本 |
| 16 | preflight-check.sh 过于严格 | 单体检查失败即退出 → 改为逐项检查 + 汇总报告模式，允许部分通过 |
