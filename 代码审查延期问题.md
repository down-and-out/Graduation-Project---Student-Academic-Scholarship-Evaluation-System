# 代码审查延期问题清单

> 来源：`/simplify` 审查（代码复用/质量/效率三轮并行），2026-05-01

以下 8 个问题因涉及跨模块重构、接口重新设计或测试框架改造，需要架构级变更，未在本次审查中直接修复，留待后续专项处理。

## 1. WebUtil.getClientIp() 未替换已有重复实现

**严重度**：中 | **类别**：代码复用

**现状**：`WebUtil.getClientIp()` 已创建并被 `OperationLogAspect` 调用，但以下两处仍保留内联的相同实现：

| 位置 | 文件 | 可见性 |
|------|------|--------|
| 新建 | `common/support/WebUtil.java:24-36` | public static |
| 已有 | `controller/AuthController.java:151-163` | private |
| 已有 | `config/EndpointRateLimitFilter.java:117-129` | private |

三处实现逻辑完全相同：`X-Forwarded-For` → `X-Real-IP` → `getRemoteAddr()`，处理逗号分隔代理链，兜底 "unknown"。

**建议方案**：将 `AuthController.getClientIp()` 和 `EndpointRateLimitFilter.extractClientIp()` 替换为 `WebUtil.getClientIp()`。

**风险**：低。`WebUtil` 是纯静态工具方法，无副作用。

## 2. Redisson 分布式锁模式分散在 5 个 ServiceImpl

**严重度**：中 | **类别**：代码复用

**现状**：以下 5 处位置各自内联了相同的 tryLock/finally/unlock 模板（每处约 10-15 行）：

| 文件 | 锁调用方式 |
|------|-----------|
| `BatchImportServiceImpl.java` | `tryLock(0, 30s)` |
| `ScholarshipApplicationServiceImpl.java` | `tryLock(0, configValue)` |
| `EvaluationTaskServiceImpl.java`（两处） | `tryLock(0, configValue)` + `lock()` |
| `EvaluationWorkflowServiceImpl.java` | `tryLock()` 无超时 |

所有实现遵循相同模板：
```java
RLock lock = redissonClient.getLock(key);
boolean locked;
try {
    locked = lock.tryLock(waitTime, leaseTime, TimeUnit.SECONDS);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
    throw new BusinessException("...");
}
if (!locked) { throw new BusinessException("..."); }
try {
    // 业务逻辑
} finally {
    if (lock.isHeldByCurrentThread()) { lock.unlock(); }
}
```

**建议方案**：抽取 `RedissonLockSupport` 工具类，封装函数式接口：
```java
public class RedissonLockSupport {
    public static <T> T executeWithTryLock(RLock lock, long waitTime, long leaseTime,
                                            TimeUnit unit, Supplier<T> action);
    public static void executeWithTryLock(RLock lock, long waitTime, long leaseTime,
                                           TimeUnit unit, Runnable action);
}
```
替代已删除的旧版 `RedisLockSupport`（手写 SETNX+Lua 版本）。

**风险**：中。需统一 5 个调用点的异常处理和锁超时配置，回归测试工作量较大。

## 3. BatchImportController 异步任务与 ExportTaskService 逻辑重复

**严重度**：中 | **类别**：代码复用 / 抽象泄漏

**现状**：`BatchImportController` 在 Controller 层直接管理异步导入任务（生成 taskId、写 Redis 状态、提交线程池、异常标记），与 `ExportTaskServiceImpl` 的任务管理能力完全平行：

| 功能 | BatchImportController | ExportTaskServiceImpl |
|------|----------------------|-----------------------|
| 任务状态存储 | `import:task:` + Redis Hash | `export:task:` + Redis Hash |
| 序列化 | `new ObjectMapper()` 每次创建 | 成员变量复用 |
| TTL | 30 分钟 | 30 分钟 |
| 状态查/写 | Controller 私有方法 | Service 层公开方法 |

**建议方案**：将 `ExportTaskService` 泛化为 `AsyncTaskService`，统一支持导出和导入两种任务类型。`BatchImportController` 应删除内联的 `writeImportTaskState`/`readImportTaskState`，通过 Service 层管理任务生命周期。

**风险**：中。涉及 `ExportTaskService` 接口变更，两个导出 Controller 也需同步调整。

## 4. ExportTaskService 接口暴露内部状态变更方法

**严重度**：高 | **类别**：抽象泄漏

**现状**：`ExportTaskService` 接口同时暴露了面向调用方的方法和面向回调内部的方法：

| 面向调用方 | 面向内部（不应公开） |
|-----------|---------------------|
| `submit()` | `markCompleted()` |
| `getStatus()` | `markFailed()` |
| `getFilePath()` | |

任何拿到 `ExportTaskService` 引用的代码都可以调用 `markCompleted(taskId, "fake.xlsx", "/etc/passwd")`，伪造任务完成状态。

**建议方案**：将 `submit()` 的 `Consumer<String>` 回调改为 `Consumer<TaskContext>`，其中 `TaskContext` 是仅暴露 `markCompleted()`/`markFailed()` 的内部接口：
```java
interface TaskCallback {
    void execute(TaskContext ctx) throws Exception;
}
interface TaskContext {
    void markCompleted(String fileName, String filePath);
    void markFailed(String errorMsg);
}
```

**风险**：中。接口签名变更影响所有 Controller 层调用方。

## 5. 两个 Controller 的异步导出 Excel 写入逻辑完全重复

**严重度**：高 | **类别**：复制粘贴

**现状**：`CourseScoreController.exportAsync` 和 `EvaluationResultController.exportAsync` 的 lambda 回调中存在完全一致的代码块：

| 逻辑 | 两处相同 |
|------|:--:|
| 临时目录构建 `System.getProperty("java.io.tmpdir") + ...` | 完全一致 |
| `new FileOutputStream(filePath)` + `EasyExcel.write()` | 完全一致 |
| `exportTaskService.markCompleted(id, fileName, filePath)` | 完全一致 |
| catch + `markFailed` 异常处理 | 完全一致 |

**建议方案**：
1. 临时目录路径抽取到 `ScholarshipProperties` 配置类中
2. 在 `ExportTaskServiceImpl` 中提供 `writeExcelFile(taskId, data, voClass, sheetName)` 工具方法，封装 EasyExcel 写入 + 目录创建 + 状态标记
3. Controller 的 lambda 只负责数据查询和 VO 转换

**风险**：低。纯代码移动，不改变行为。

## 6. importScores / 导出无分页，大数据量 OOM

**严重度**：高 | **类别**：内存安全

**现状**：

**a) `CourseScoreServiceImpl.importScores()`**
- `newScores` 和 `updateScores` 两个 ArrayList 全量驻留内存
- 循环结束后才调用 `persistImportResults` 批量写库
- 如果上传 50,000 行的 Excel，两个列表同时持有 50,000 个对象

**b) 导出**（`CourseScoreController` 和 `EvaluationResultController`）
- 异步导出的回调内部调用无分页的 `list()` 或 `exportBatchResults()`
- 全量加载到内存后一次性写入 EasyExcel
- 10 万+ 条记录时 JVM 内存可能打满

**建议方案**：
- **导入**：参照 `BatchImportServiceImpl` 的 `BATCH_FLUSH_SIZE = 500` 模式，在循环中当累积量达到阈值时立即调用 `self.persistImportResults()` 分批写库并 clear
- **导出**：改为分页游标式处理，每次查询 500 条，循环写入 EasyExcel 的流式输出

**风险**：中。导入分批刷新需要确保事务边界正确（当前 persistImportResults 在独立事务中，分批调用没有问题）。

## 7. BatchImportServiceImpl 分批提交无整体事务保护

**严重度**：高 | **类别**：数据一致性

**现状**：`importStudents()` 方法无 `@Transactional`，`studentInfoService.saveBatch(toInsert, 200)` 每 500 条独立提交。如果第 501 条记录违反唯一约束，前 500 条已提交的数据不会回滚。

**建议方案**（三选一）：
- **方案 A**：加 `@Transactional` 并将分批提交改为全量一次性写入（大事务有性能风险）
- **方案 B**：在内存中先做全量校验（学号去重、数据合法性），确认通过后再在事务内一次性写入
- **方案 C**（推荐）：失败时通过补偿机制（按时间戳/批次号标记）删除已导入的同批次数据

**风险**：高。直接加 `@Transactional` 可能导致大事务锁表，推荐方案 B 或 C。

## 8. @Lazy @Autowired self 代理注入在纯 Mockito 测试中为 null

**严重度**：高 | **类别**：测试兼容性

**现状**：`CourseScoreServiceImpl` 和 `SysNotificationServiceImpl` 使用了 `@Lazy @Autowired private Xxx self` 模式。

当测试使用 `@ExtendWith(MockitoExtension.class)` + `@InjectMocks` 时：
- `@InjectMocks` 不处理 `@Autowired` / `@Lazy` 注解
- `self` 字段保持 null
- 调用 `self.xxx()` 的方法会抛出 `NullPointerException`

**受影响文件**：
- `CourseScoreServiceImpl` — `importScores()` 末尾调用 `self.persistImportResults()`
- `SysNotificationServiceImpl` — `sendNotificationAsync()` 调用 `self.sendNotification()`

**建议方案**：
- **方案 A**（推荐）：在 `@BeforeEach` 中使用 `ReflectionTestUtils.setField(service, "self", service)` 手动注入
- **方案 B**：将 `persistImportResults` / `sendNotification` 提取为独立的内部 Service，走常规构造器注入
- **方案 C**：使用 `@SpringBootTest` 集成测试替代纯 Mockito 单元测试

**风险**：低（仅影响测试代码，不影响生产运行）。
