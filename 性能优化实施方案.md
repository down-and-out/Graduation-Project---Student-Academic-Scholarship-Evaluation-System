# 研究生学业奖学金评定系统 — 性能优化实施方案

## Context

当前系统存在 8 个性能瓶颈（4 个严重、4 个高），主要集中在：长事务持有数据库连接、数据重复加载、同步导出无分页、异步基础设施未充分利用、操作日志写入完全缺失。本方案按 5 个阶段顺序执行，从基础设施补齐到业务优化再到定时清理，逐步推进，每阶段可独立验证。

### 实施进度

| 阶段 | 步骤数 | 状态 | 备注 |
|------|:------:|:----:|------|
| 阶段一 | 4 | ✅ 已完成 | 基础设施搭建 |
| 阶段二 | 4 | ✅ 已完成 | SQL/事务优化 |
| 审查修复 | 4 | ✅ 已完成 | 删除 RedisLockSupport、WebUtil、tryLock、复用 buildExecutor |
| 阶段三 | 4 | ✅ 已完成 | 异步迁移（共享 ExportTaskService 替代各模块独立 Service） |
| 阶段四 | 1 | ✅ 已完成 | Druid delete-permit 修复 |
| 阶段五 | 2 | ✅ 已完成 | Token 计数器校准 + 日志清理 |

> 全部 19 个步骤已落地：新建 7 个文件，修改 21 个文件，删除 1 个文件（`RedisLockSupport`）。

## 阶段一：基础设施搭建 (风险：低)

**目标**：补齐异步/定时/AOP 基础能力，为后续所有优化提供支撑。

### ✅ 步骤 1.1 — 启用 @EnableScheduling

**文件**：`scholarship-admin/src/main/java/com/scholarship/ScholarshipApplication.java:40`

在 `@EnableAsync` 下方添加 `@EnableScheduling`：

```java
@EnableAsync
@EnableScheduling
```

> 仅启用 Spring 调度能力，不添加任何定时任务，零副作用。

### ✅ 步骤 1.2 — AsyncTaskConfig 实现 AsyncConfigurer 接口

**文件**：`scholarship-admin/src/main/java/com/scholarship/config/AsyncTaskConfig.java`

- 类声明改为 `public class AsyncTaskConfig implements AsyncConfigurer`
- 覆盖 `getAsyncExecutor()`：返回默认线程池（core=2, max=4, queue=50, CallerRunsPolicy）
- 覆盖 `getAsyncUncaughtExceptionHandler()`：记录异步方法异常日志
- 新增 import：`org.springframework.scheduling.annotation.AsyncConfigurer`、`org.springframework.aop.interceptor.AsyncUncaughtExceptionHandler`、`java.util.Arrays`

> 确保 `@Async` 无参使用时也有合理线程池，且异常不被静默吞掉。

### ✅ 步骤 1.3 — 创建 @OperationLog 注解

**新建文件**：`scholarship-admin/src/main/java/com/scholarship/common/annotation/OperationLog.java`

核心字段：
- `module()` — 操作模块，默认空
- `type()` — 操作类型，`OperationLogTypeEnum`
- `description()` — 操作描述
- `logParams()` — 是否记录请求参数，默认 true
- `logResult()` — 是否记录响应结果，默认 false

> 纯注解定义，不影响现有代码。

### ✅ 步骤 1.4 — 创建 OperationLogAspect 切面

**新建文件**：`scholarship-admin/src/main/java/com/scholarship/common/aspect/OperationLogAspect.java`

- 依赖注入：`SysOperationLogMapper`（已存在）
- `@Around("@annotation(opLog)")` 拦截
- 从 `SecurityContextHolder` 获取当前用户信息
- 从 `RequestContextHolder` 获取 `HttpServletRequest`（IP、URL、Method）
- 记录 `executionTime`（start → end 耗时）
- **使用 `@Async` 异步写库**，不阻塞业务线程
- 异常时也要记录日志（status=0, errorMsg），然后 rethrow

> 新建文件，注解需主动添加才生效。依赖 1.3。

### 阶段一验证
- 启动应用，检查日志无异常
- 给任意 Controller 方法加 `@OperationLog(type = OperationLogTypeEnum.SYSTEM, module = "测试")`，调用后查 `sys_operation_log` 表确认写入成功

## 阶段二：SQL/事务优化 (风险：中)

**目标**：修复 4 个严重瓶颈的不良模式，不引入异步。

### ✅ 步骤 2.1 — EvaluationWorkflowServiceImpl 数据复用

**文件**：`scholarship-admin/src/main/java/com/scholarship/service/impl/EvaluationWorkflowServiceImpl.java`

**当前问题**：`evaluateBatch()` 中 `generateBatchRanks(batchId)` 和 `allocateAwards(batchId)` 各自执行 `selectList` 全量加载 `EvaluationResult`，数据重复加载 2 次。

**修改**：
1. 新增注入 `EvaluationResultMapper evaluationResultMapper`
2. 在 `calculateBatchApplications` 之后、`generateBatchRanks` 之前，统一执行一次 `selectList` 加载全部结果（按 `totalScore` 降序）
3. 将第一步、第二步的单参重载方法（`generateBatchRanks(Long, List)` 和 `allocateAwards(Long, List)`）替换原来的单参调用
4. 逻辑与 `EvaluationTaskServiceImpl.doExecuteTask():174-181` 已有的成熟实现完全一致

> 减少 2 次全表扫描，直接从已加载的 List 传入排名和分配阶段。风险低，异步路径已验证。

### ✅ 步骤 2.2 — BatchImportServiceImpl 分布式锁 + 事务缩小

**修改文件**：
- `scholarship-admin/src/main/java/com/scholarship/common/support/LockConstants.java` — 新增 `BATCH_IMPORT_STUDENTS = "lock:batch:import:students"`
- `scholarship-admin/src/main/java/com/scholarship/service/impl/BatchImportServiceImpl.java`

**当前问题**：整个 `importStudents()` 方法被 `@Transactional` 包裹，无分布式锁，并发导入有 TOCTOU 风险。

**修改**：
1. `importStudents()` 移除 `@Transactional` 注解
2. 注入 `RedissonClient redissonClient`
3. 方法开头通过 Redisson 获取分布式锁（`lock.lock(30, TimeUnit.SECONDS)`）
4. 仅 `saveBatch()` 保留事务（MyBatis-Plus 内置事务保护）
5. finally 释放锁

> 事务只包裹 DB 写入，锁保证并发安全。

### ✅ 步骤 2.3 — CourseScoreServiceImpl 事务缩小

**文件**：`scholarship-admin/src/main/java/com/scholarship/service/impl/CourseScoreServiceImpl.java`

**当前问题**：`importScores()` 的 `@Transactional` 包裹 Excel 解析 + 逐行处理 + 缓存加载 + 批量写入，事务持有时间过长。

**修改**：
1. 将 `importScores()` 上的 `@Transactional` 移除
2. 抽取 `private void persistImportResults(List<CourseScore> newScores, List<CourseScore> updateScores, List<Long> dupIds)` 方法，仅此方法带 `@Transactional`
3. 注入自身代理：`@Lazy @Autowired private CourseScoreServiceImpl self`
4. 通过 `self.persistImportResults(...)` 调用使事务生效

> Excel 解析（纯内存）不再占用数据库连接和事务。

### ✅ 步骤 2.4 — ScholarshipApplicationServiceImpl 迁移到 Redisson

**文件**：`scholarship-admin/src/main/java/com/scholarship/service/impl/ScholarshipApplicationServiceImpl.java`

**当前问题**：`submitApplication()` 使用已标记 `@Deprecated` 的 `RedisLockSupport`（手写 SETNX+Lua），项目其他模块已统一使用 Redisson。

**修改**：
1. 注入 `RedissonClient redissonClient`（替换 `RedisLockSupport`）
2. 锁获取改为 `lock.tryLock(0, ttlSeconds, TimeUnit.SECONDS)`
3. 锁释放改为 `lock.isHeldByCurrentThread()` + `lock.unlock()`
4. 移除 `UUID` 生成（Redisson 内部管理锁标识）
5. 移除 `RedisLockSupport redisLockSupport` 注入

> 锁语义一致，技术栈统一。后续可全局搜索确认无调用方后删除 `RedisLockSupport`。

## 阶段三：异步迁移 (风险：中) ✅ 已完成

**目标**：利用已有的 `exportTaskExecutor` 和 `batchImportTaskExecutor` 线程池，将重操作从 HTTP 线程剥离。

**实施说明**：创建了共享的 `ExportTaskService` / `ExportTaskServiceImpl`（而非方案中最初设计的各模块独立 Service），统一管理导出任务生命周期，两个导出 Controller 复用同一基础设施。

### ✅ 步骤 3.1 — 成绩导出异步化

**新建文件**：
- `scholarship-admin/src/main/java/com/scholarship/service/ExportTaskService.java`
- `scholarship-admin/src/main/java/com/scholarship/service/impl/ExportTaskServiceImpl.java`

**修改文件**：
- `scholarship-admin/src/main/java/com/scholarship/controller/CourseScoreController.java`

**异步导出流程**：
```
POST /course-score/export/submit  →  返回 {"taskId": "uuid"}
GET  /course-score/export/status/{taskId}  →  {"status": "PROCESSING|COMPLETED|FAILED", "filePath": "..."}
GET  /course-score/export/download/{taskId}  →  Response（文件流）
```

**实现要点**：
- `ExportTaskServiceImpl` 使用 `exportTaskExecutor` 异步执行，Consumer<String> 回调模式传入 taskId
- 使用 `StringRedisTemplate` 存储导出任务状态（Key: `export:task:{taskId}`，TTL 30 分钟）
- EasyExcel 流式写入临时文件到 `java.io.tmpdir/scholarship-exports/`
- 前端按 1 秒间隔轮询状态，完成后用 `window.open` 触发下载

> 保留原同步导出端点 `/export` 不删除，作为降级方案。
